# proofread

現在の git ブランチと同名のフォルダ内にある `article.md` を校正する。このリポジトリでは「1 記事 = 1 ブランチ、ブランチ名 = フォルダ名」という規約（README.md 参照）。

## 手順

### 1. 対象ファイルの特定

1. `git branch --show-current` で現在のブランチ名を取得
2. `<branch-name>/article.md` を読む
3. ブランチが `main` や該当フォルダが存在しない場合は、ユーザーに対象ファイルを確認

### 2. チェック観点

以下の観点で記事全体を走査し、問題箇所を **該当箇所の引用 + 指摘 + 修正提案** の形で列挙する。行番号（`article.md:42` 形式）も付ける。

#### (A) 誤字脱字
- タイプミス、変換ミス、文字化け
- 助詞の誤り（「てにをは」）
- 送り仮名の誤り

#### (B) 表記ゆれ
- カタカナ語の揺れ（例: 「サーバ」と「サーバー」が混在）
- 英数字の全角/半角の混在
- 用語表記の統一（例: 「Unity」「unity」「ユニティ」）
- 記事内で一貫していれば OK、混在している場合は統一提案

#### (C) 技術用語の正確性
- 用語の誤用・不正確な使い方（例: 「コンパイル」と「ビルド」の混同）
- API 名、クラス名、メソッド名、プロパティ名の綴り
- バージョン番号の表記
- コードブロック内のシンタックス誤り

#### (D) 読みやすさ
- 一文が長すぎる（目安: 80 字超）
- 主語と述語のねじれ
- 受動態/能動態の不自然な混在
- 同じ語尾の連続（「です」「です」「です」）
- 段落が長すぎる／見出し構造が崩れている

#### (E) 説明の事実確認（重要）
以下の記述は **必ず検証** する:

- **API の仕様・シグネチャ**: 公式ドキュメント（WebFetch / WebSearch）で確認
- **ライブラリ・フレームワークのバージョン固有の挙動**: 当該バージョンのドキュメントで確認
- **コード例**: 実際に動作する書き方か。存在しないメソッドや引数を使っていないか
- **手順**: 記載されたコマンドや UI 操作が実際に機能するか（入手可能な情報の範囲で）
- **数値・統計**: 出典のない数字は疑う
- **断定表現**: 「必ず」「絶対」「〜しかない」等の強い主張は根拠を確認

嘘・誤り・古い情報を発見したら **優先度: 高** として明記する。確認が取れないものは「要確認」として列挙し、勝手に「正しい」と判断しない。

### 3. 出力フォーマット

```
## 校正結果: <branch-name>/article.md

### 事実誤認の可能性（優先度: 高）
- `article.md:42` 「〜は〜である」
  - 指摘: 公式ドキュメント（URL）によると実際は〜
  - 修正案: 〜

### 誤字脱字
- `article.md:10` 「〜」→「〜」

### 表記ゆれ
- 「サーバ」(行 12, 45) と「サーバー」(行 78) が混在 → どちらかに統一推奨

### 技術用語
- ...

### 読みやすさ
- ...

### 要確認（裏付けが取れなかった記述）
- ...
```

### 4. 修正適用

指摘を提示した後、ユーザーが修正適用を希望した場合のみ Edit で反映する。勝手に書き換えない。

## 注意

- frontmatter（`---` で囲まれた YAML）は校正対象外。ただし `title` の誤字は指摘する
- Obsidian 記法の画像参照 `![[./images/xxx.png]]` や `![[./images/xxx.png|400]]` は記法として正しいので指摘しない（README.md 参照）
- コードブロック内の文字列はコード文法に従って評価する（日本語文法の指摘対象外）
- 問題がなかった観点は「問題なし」と一言で報告する（無理に指摘を捻出しない）
